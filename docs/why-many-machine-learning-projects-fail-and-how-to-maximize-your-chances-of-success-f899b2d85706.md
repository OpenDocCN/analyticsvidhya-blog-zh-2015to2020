机器学习和数据科学(ML/ DS)是当今的一个时髦词，许多组织，特别是软件和技术领域的组织，正在通过创建数据科学团队和实施一些使用 ML 算法的项目来投资这一领域。

我在某处读到过，大约 80%的机器学习项目失败了，因为没有成为最终产品的一部分。许多这样的失败发生在不熟悉机器学习(ML)或不熟悉 ML 的组织或团队试图在没有做足够功课的情况下实现 ML 想法，并期望它成功并立即给出更好的结果的地方。

在这篇文章中，基于我在一些机器学习项目中的工作经验，我想提出一些可能导致 ML 项目失败的因素，以及可以做些什么来减轻这些因素。我的希望是，对这些因素的分析和认识将使这种错误不太可能重复。

## **因素 1:不明确的预期**

ML 或数据科学项目的本质是它们遭受太多的变量和不确定性。与普通的软件工程项目不同，ML 项目在第一次尝试时可能会也可能不会给出更好的性能。他们也有点像黑箱。ML 解决方案没有给出更好的结果可能有多种原因:模型选择错误、过拟合或欠拟合、缺乏交叉验证、变量或参数太多以及缺乏网格搜索以找到特定问题的最佳参数。

许多团队被宣传所迷惑，期望任何 ML 模型都比当前的解决方案表现得更好，而当前的解决方案可能已经调整了很多年。此外，不能保证任何 ML 解决方案会比基于规则的解决方案执行得更好。这完全取决于要解决的问题的性质。

了解了所有这些之后，更好的前进方式是保持较低的期望值，并专注于相对快速地获得一个工作原型模型，最好是进入一个测试或生产环境，在那里可以获得真实的数据来进一步调整模型。该模型的准确性可以在以后改进。渐进式方法，在模型的每个连续版本中都有许多小的成功，通常比大跃进方法更有可能成功，因为大跃进方法在投入生产之前要花大量时间进行开发和模型调整。

一旦基础设施到位，改变算法或 ML 模型就相对容易了。同样有用的是，首先花大量的时间研究和精确定义预期的结果，这些结果是基于其他人在解决相同问题时的表现的可用数据。一种方法是查看该问题领域的最新论文(例如来自 arxiv.org 的论文),并尝试在论文中复制结果。

## **因素 2:缺乏合格的团队**

许多刚接触 ML 的组织犯了一个错误，就是在开始 ML 工作之前没有确保他们有一个合格的团队。一个 ML 解决方案的工作，需要有适当的支持和专业知识。为此，你必须雇佣或培训一个有 ML 项目经验的团队。这需要团队和大量的投资，组织应该为失败做好预算，给这样的团队足够的时间来建立。

然而，对于一个第一次涉足这个领域的组织来说，这可能是一个先有鸡还是先有蛋的问题。这方面的人才需求量很大，而有技能的人才可能很难找到。没有好的团队，就不会有结果，如果最初的结果不够好，许多组织就会推迟投资。有一个了解 ML 的经理对一个团队的成功很重要。

任何团队或组织都应该小心的一个错误是雇佣一些 ML 专家，并期望他们在没有提供足够支持的情况下交付最先进的模型。这里要记住的一件重要事情是，ML 包括各种专业领域，如图像处理、计算机视觉、自然语言处理、流数据处理等，每一个领域都需要不同类型的技能。

此外，一个好的数据科学家可能不是一个好的软件工程师，反之亦然，可能需要帮助和时间来理解现有的代码库和集成他们的解决方案，并且可能不具有他们应用 ML 解决方案的系统的良好领域知识。了解了所有这些，一个组织应该确保足够的支持(以主题专家和软件工程师的形式)对 ML 工程师执行他们的工作是可用的。

ML 工程师或数据科学家也需要支持和持续的学习来保持在他们的游戏中。这种支持可能包括使 ML 团队能够在会议中与 ML 专家和领导者互动，赞助和鼓励参与 ML 黑客马拉松和 kaggle 类竞赛，通过阅读论文了解最新模型等。对于希望对现有员工进行内部培训的组织来说，一种方法是让他们参与一个或多个现实生活中的 ML 项目(不仅仅是一项学术任务)，并让他们接受 ML 专家的监督。

## **因素 3:范围蔓延**

许多 ML 项目遭受范围蔓延，ML 项目的目标不断变化。这可能意味着不同的问题:一个是组织没有很好地定义或理解目标，特别是如果它没有 ML 项目的经验作为基准。有模糊的需求并经常在项目进行中改变它们通常不是一个好的迹象。

敏捷方法通常在这方面有所帮助，在每个 sprint 中都要完成明确定义的工作。重要的是将复杂的 ML 问题分解成小步骤，并致力于首先将最简单的端到端解决方案投入生产。有一种解决方案可能会有所帮助，其中 ML 模型输出与非 ML 输出相结合，并且采用输出的集合。在 ML 和非 ML 解决方案之间进行切换，并在解决方案被允许投入生产之前，对 ML 解决方案和非 ML 解决方案进行适当的 A/B 测试，这也可能有所帮助。

## **因素 4:缺乏数据，以及创建可用训练数据的机制**

对于 ML 模型来说，数据就像氧气，因此首先构建机制以获取数据非常重要。有人说好的数据甚至比好的算法更重要。大量的好数据不容易获得，而且可以获得的数据可能都没有标注。这个问题因问题而异，数据集可用于某些类型的问题，但不可用于其他类型的问题。

另一件重要的事情是确保衡量结果的机制到位。这包括衡量成功的参数和指标，包括精确度和召回率、用户度量、用户满意度调查等。没有这一点，就没有办法知道一个模型的表现有多好。这可能包括在模型投入生产后查询用户数据的机制。前几周花时间来确保参数和期望被正确定义是非常值得的。

解决方案可能包括首先构建基础架构以获得良好的数据，例如让团队标记一些数据，并使用半监督方法根据一些最初标记的小数据集自动标记大量数据。

## **因素五:信任缺口**

这可能是一些 ML 解决方案无法投入生产的另一个因素。有时，想要投资 ML 的团队或组织有自己一套既定的工作方式，可能会对 ML 团队推荐的解决方案产生抵触或感到威胁。有时，尽管经理们希望投资于 ML，但还是存在信任缺口。

补救的一个方法是清楚地划分期望，并确保大团队中的每个人至少熟悉更高层次的细节和 ML 的术语。

## **因素 6:如果项目失败，缺乏适当的事后分析和分析**

一些团队犯的一个错误是没有花时间去正确地分析为什么一个 ML 投资失败或者一个项目不能进入生产。每一次失败都有宝贵的经验，不从自己的失败中吸取教训就是浪费重要的数据，而这些数据对未来项目的成功是有价值的。

鉴于许多这样的 ML 项目由于上述原因或其他原因的组合而失败，如果最初的几项投资失败，这并不是完全出乎意料的。因此，如果项目失败了，重要的是要深入思考每一步，是否犯了错误以及如何减少错误(如果有的话)。

**结论**

总之，任何新寻求进入 ML 领域的团队或组织都应该确保他们的方法没有提到的标志，并且他们有一个得到良好支持的 ML 团队，有明确定义和现实的期望，以多个版本递增地交付 ML 解决方案。
