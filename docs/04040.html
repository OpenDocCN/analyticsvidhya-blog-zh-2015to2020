重新评估纸上解决方案。

本文是关注使用分析解决常见业务问题的系列文章的第一篇。对于第一个问题，即工作负载评估，我决定分成五个较小的部分，这样我就可以更详细地讨论这个问题，并希望为解决这类问题提供一个有用的例子。这五个会议将是: 将业务问题转化为分析问题 ，重新评估纸上解决方案(当前会议)， 生成并加载测试数据 ，提出解决方案，最后是项目总结和演示。

对你的假设进行一致的批判性重新评估是分析过程中最重要的行动之一。如果不愿意撕毁你投入精力的潜在解决方案，你的产出将会受损，而且存活的可能性很小。你必须愿意在这个过程中的任何一个点上以极度的小心和严谨去做这件事。你最不希望发生的事情就是到达生产阶段，在你的逻辑中发现一个站不住脚的漏洞。为了避免这种噩梦，尽可能多的用批判的眼光来看待它。

拆除:

在上一次会议中，我提出了我认为这个问题的解决方案。经过重新评估，我认为这个问题并没有完全解决。我也认为我对这个问题挖掘得不够深入，没有完全分解到它的核心部分。让我们回顾一下上一节课。

前一届会议的成果可总结如下:

Volume Coefficient = ∑((μ−x)/σ)÷n
Magnitude = Time * (∑((μ−x)/σ)÷n * {max⁡(r)│Category})
Capacity = FTE * (Time * (∑((μ−x)/σ)÷n * {max⁡(r)│Category}))

我认为这背后的总体想法是有效的，但执行需要一些工作。我们正在创建一个标准化的体积测量，并对任务的难度进行加权，然后乘以我们拥有的员工数量。

首先，让我们看看我们的标准化体积测量。

Volume Coefficient = ∑((μ−x)/σ)÷n

这个测量基本上是告诉你你的体积测量值相对于它们的标准偏差平均离它们的总体均值有多远。这真的是我们想要完成的吗？这将使这些体积测量值的权重相等。我认为这不符合我们的既定目标。为了实现我们的目标，我们需要恰当地权衡这些度量的重要性和价值。我们还没有完成标准化，所以这里的既定目标肯定没有实现。

接下来，让我们评估幅度函数:

Magnitude = Time * (∑((μ−x)/σ)÷n * {max⁡(r)│Category})

我们的时间变量是这里的难度调整，然后我们进行音量归一化，并乘以所选的音量类别。这里要注意的第一件事是，r应该是 ρ ，因为我们在看总体与样本皮尔逊相关系数。另一个突出的问题与我们的音量正常化有关。我们正在决定使用哪种测量方法，但这种选择不会影响标准化，因此目标没有实现。

最后，我们的容量评估:

Capacity = FTE * (Time * (∑((μ−x)/σ)÷n * {max⁡(r)│Category}))

这里的想法是将单个资源可以处理的容量乘以我们的可用资源。这里也没有完成。这里的问题是少了一个步骤。我们没有对单个资源的预期贡献进行评估，因此这个目标在这里也无法实现。

现在这些假设已经被彻底推翻了，让我们来做一些新的假设。

重新评估:

让我们重新开始，花点时间回顾一下原来的问题，重新定义范围。

业务问题:没有足够的资源来处理当前的工作量。

为了做出这一决定，您需要回答以下问题:

  • 你如何衡量工作？
  • 工作总量是多少？
  • 单个资源可以处理多少工作负载？
  • 我们目前的资源能处理多少工作？
  • 我们需要多少员工来完成我们的要求？

我们需要解决的核心问题是:你如何衡量工作？我们需要回答的其他问题都需要先回答这个问题。因此，让我们逐一分析这些问题，并尝试为每个问题提出可行的解决方案。

你如何衡量工作？

这是你需要开始研究的地方。你可能已经从项目发起人那里获得了一个坚实的起点，但是你需要采访各个层面的利益相关者来收集尽可能多的信息。经理衡量工作量的方式可能不同于层级中每个级别的员工，每个级别中的人衡量工作量的方式也可能不同。不要有太多的特性，但是你会想要收集大量的特性来进行适当的评估。这将防止持怀疑态度的利益相关者在你进入陈述阶段时用 whataboutisms 毒害你的分析。这些特征中的每一个都必须是体积测量的某种变化。在我们的例子中，我们将使用采购计数、已开发票和采购金额作为我们的数量度量。

我们的总工作量是多少？

好了，我们现在有一些潜在的度量，那么我们如何知道选择哪一个呢？这是很多错误发生的地方；与你交谈过的大多数利益相关者都有一个通用的规则，但也有一些例外。这是很好的信息，将在您稍后测试您的分析时提供洞察力。一般来说，一刀切的方法是行不通的。如果是这样的话，你可能就不会负责这个问题了。

那么，对于给定的场景，我们如何选择最佳的测量方法呢？

这里是你的研究员帽子回去的地方:你必须识别重要的分类特征。你在哪里找到这些特征的？首先是在你的利益相关者访谈中，你应该对这些讨论有一个很好的想法。您应该在最初的分析中尽可能多地包括您可以访问的可用功能。在这一点上有额外功能方面的错误，在未来我们可以放弃那些没有增加价值的功能。对于我们的例子，我们将使用业务单位、客户和供应商作为我们的特征。

既然我们已经有了我们的特性和潜在的度量，那么我们如何确定对一组特性使用哪种度量呢？这就是事情开始变得(*咳，相对来说，咳*) 有趣的地方。本质上，总功量是一个恒定的线性函数。这是一个没有斜率的线性方程，也叫水平线。

total_volume = ∑(volume_units)

好吧，那又怎样？为什么这很重要？这象征着我们已经知道的东西:在真空中，这些不同的体积测量值都有相同的影响。每个不同的体积测量值本身是一条简单的水平线。这告诉我们，我们的体积测量只有在与另一个变量相关时才有意义，因此，没有另一个变量，我们就没有基础来做出测量选择。在我们探索剩下的问题时，请记住这一点。

单个资源可以处理多少工作负载？

如何衡量单个资源可以合理处理多少工作？换句话说，单个员工的预期贡献量是多少？前面有几条路。一种，也可能是最常见的一种，是用我们在此期间拥有的员工人数对总完成工作量进行简单平均。

avg_volume = avg(total_volume)/employee

然而，一个简单的平均值并不能真正洞察完成一个单位工作量需要多长时间或多少工作量。这很重要，因为我们需要能够平衡一个工作单元对另一个工作单元的挑战。如果一种工作在数量上比另一种更具挑战性，我们需要能够考虑到这一点。这是你需要深入挖掘以找到最佳解决方案的情况之一。这个故事还有更多的内容，如果你就此打住，你就失去了宝贵的洞察力。同样，在你的利益相关者不喜欢某个结果的第一个迹象时，这个疏忽将被用来削弱你的分析的有效性以及你的可信度。这个故事的寓意:深入挖掘，尊重问题。

因此，如果简单的平均值不行，我们该何去何从？我们需要评估与不同类型的工作相关的挑战。我们需要一种方法来衡量完成一个工作单元所需要的努力。我们如何衡量一个人的努力？这是一个极具挑战性的问题，有大量嘈杂的变量在起作用，其中大多数我们甚至无法有效地测量。每个人每天都面临着不同的个人和职业挑战，每个员工都有不同的能力，所以即使完全相同的任务也需要不同的努力来完成。我们如何衡量不同人完成不同任务的努力程度？呀。

这就是你要开始考虑组合变量的地方。我们在测量工作中遇到的每一个问题都需要结合起来，用一个变量来表示。如果我们思考上述每个问题，在整个人口中，每个问题都是同一个函数的一部分。方便的是，我们实际上已经知道了这个函数的结果，所以我们可以简单地用它作为我们的组合变量。我所指的函数是努力的成本。任何组织的努力成本是什么？努力的代价是时间。反对用时间来表示努力的论点类似于我们上面所阐述的挑战。相同的任务在不同的员工之间会花费不同的时间，如果员工没有付出真诚的努力，这种测量会让他们看起来像是在努力工作，即使他们并没有。虽然这些批评是正确的，但在整个人群中，这些因素应该是最小的，您将能够识别这些异常值，并且管理层可以在必要时解决它们。现在，如果一位经理告诉你，他们的整个团队都符合这一类别，提醒他们这是一个不合理的职位，如果情况确实如此，他们需要审查他们的招聘实践和管理策略。这个立场也与分析的有效性无关，因此不是一个有效的批评。

有了这种认识，让我们重新审视一下我们的预期产量贡献。

avg_volume = avg(total_volume)/employee
avg_efficiency = avg_volume/avg(time)exp_vol_contribution = avg_volume <--- avg_efficiency - σ < avg_volume < efficiency + σ

我们的平均体积测量没有改变，但我们增加了一个效率分数，即平均体积除以完成一个单位的平均时间。然后，为了得到我们的预期贡献，我们将取平均效率在平均值的标准偏差内的平均体积。这将返回员工达到我们的效率标准范围的平均工作量。这告诉我们，当一个普通员工以平均水平的效率工作时，我们应该期望他们做出什么贡献。

努力呢？

我们还没有解决努力的问题。我们现在需要做的是想出一种方法，根据完成一个具有给定的一组特征的工作单元所需的努力来惩罚或奖励我们的体积分数。我们已经确定可以用时间来衡量我们的努力程度，因此这将是我们奖惩制度的基础。为了确定我们的调整，我们可以采用我们的特征集的平均值远离总体平均值的标准偏差的数量。

effort = 1+((x-μ)/σ)

我喜欢这种调整，因为它以总体平均值为中心，并增加了数据集的分布，作为其视角的一部分。

现在我们需要的东西几乎都有了，但是房间里还有一头大象。到目前为止，我们一直在以某种形式使用体积，但我们无法完成对如何测量体积的评估，因为我们没有足够的信息。我们说过:

total_volume = ∑(volume_units)

在我们继续完成评估之前，我们需要弄清楚如何测量体积。我们确定我们的体积测量只有在与另一个变量相关时才有意义，如果没有这种关系，我们就无法评估任何测量。我们实际上已经理解了这一部分。我们要考虑的关系是与努力的关系。

relative_volume = effort * total_volume

由于这是一个线性方程，为了确定使用哪种体积测量方法，我们需要确定哪种测量方法与工作量的关系最密切。我们可以用皮尔逊相关系数来评估这一点。

ρx,y = co-variance(x,y)/(σₓ*σᵧ)

我们只需要对每组特征的每个测量值进行评估，并选择每组特征的最大值作为我们的测量值。

volume_category = max((ρEffort,vol_measure₁),(ρEffort,vol_measure₂),...)

我们将把它记为:

volume_category = {category|max{ρ(effort,vol_measure):∀vol_measure}}

我们的量类别是努力和量的相关系数是所有可用量度量的最大值的类别。

理想情况下，这将为不同的特征集提供不同的度量。这意味着在我们对这些体积测量值做任何其他事情之前，我们必须将它们标准化。

normalized_units = ∑{units/max(units)|volume_category}

这里，我们将体积单位的总和除以每个给定体积类别的最大体积单位。这将为我们提供一个相对于每个类别中最高值的数量，这些数量可以加在一起。

既然我们已经有效地解决了这个核心问题，我们现在实际上已经拥有了我们需要的所有东西来回答我们最初的问题。

总体积:

difficulty_adj = (x-μ)/σ
volume_category = {category|max{ρ(effort,vol_measure):∀vol_measure}}
normalized_units = ∑{units/max(units)|volume_category}
relative_volume = effort * normalized_units

预期贡献:

avg_volume = avg(relative_volume)/employee
avg_efficiency = avg_volume/avg(time)exp_vol_contribution = avg_volume <--- avg_efficiency - σ < avg_volume < efficiency + σ

以我们目前的资源，我们能处理多少工作？

要回答我们的工作负载能力问题，我们只需要将我们的预期贡献乘以员工数量。

capacity = exp_vol_contribution * count(employees)

我们需要多少员工来完成我们的要求？

要确定完成我们的职责需要多少贡献者，我们只需将我们的相对量除以每个员工的预期贡献。

req_contributors = relative_volume/exp_vol_contribution

离别的思念

分析生命周期需要一致的关键评估。不要害怕在任何时候撕毁你的作品。这是过程的一部分，也是进步之路。如果你每天对这个问题了解得更多，你最初的假设很可能会在没有进化的情况下继续存在。接受建设性的批评，并坚持不懈地寻找它。

我们现在有了有效回答所有原始问题的弹药，而且几乎同样重要的是，我们在快速处理其他资源相关问题方面处于有利地位。一旦我们的核心问题解决了，剩下的问题就变得非常简单了。
